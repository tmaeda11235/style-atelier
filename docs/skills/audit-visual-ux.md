# Audit Visual UX

このスキルは、あなた（AI）が「チームの目（Visual UX Auditor）」として、アプリケーションの視覚的な品質と、LLM（WebLLM / LiteRT-LM）とのインタラクション時の動的なUXを監査し、不具合を報告するための手順書です。モバイルPWA（Progressive Web App）特有の画面制約やタッチ・オフライン特性を含めた評価手法を規定します。

## 評価基準（The "Eyes"）

以下の6つの観点を**厳しく**チェックしてください。

1. **テーマの整合性（Dark/Light Mode Consistency） [最重要]**
   - ダークモードなのに背景が白くなっているコンポーネントがないか。
   - ライトモードなのに文字や背景が真っ黒で浮いているコンポーネントがないか。
   - Tailwindのクラス指定漏れやハードコードされた色が原因で、テーマ切り替え時に視認性が損なわれていないかを徹底的に探します。

2. **視認性とコントラスト（Legibility & Contrast）**
   - 背景色と同化して文字が読めなくなっていないか。
   - テキストがコンテナから溢れていたり、他のアイコンやボタンと重なっていないか。

3. **LLMの動的レンダリング（Dynamic Rendering of LLM Streaming）**
   - LLMからのストリーミング応答時、テキストがスムーズに表示されているか。
   - Markdownのコードブロックやテーブルが崩れず、正しくスタイルが適用されてレンダリングされているか。
   - テキストの生成に合わせてオートスクロールが自然に機能しているか。

4. **ダウンロード時の不安感緩和（Download Anxiety UX）**
   - LLMモデルの初回ダウンロード時、プログレスバーやローディング表示が適切な位置で、適切なサイズで表示されているか。
   - ユーザーが「フリーズしたのでは？」と不安にならないような視覚的フィードバックが提供されているか。

5. **可観測性とロギング（Observability & Logging）**
   - Playwrightのトレースやブラウザのコンソールログを確認し、重要なユーザーアクションやLLMの状態遷移が適切に記録されているか。
   - 視覚的バグやUX摩擦が発生した際に、原因究明に役立つ十分なログが出力されていない場合、「ログ仕込み漏れ」として検知します。

6. **モバイルPWA特有のUI/UX（Mobile PWA & Touch UX）**
   - **セーフエリア対応**: iOSのノッチ（カメラの切り欠き）や下部のホームインジケーター等のデバイス固有セーフエリア（`safe-area-inset-*`）を意識した余白が確保され、UI要素（ヘッダーや操作ボタンなど）が隠れていないか。
   - **タッチ操作の最適化**: ボタンやバインダーのタブなどのタップ可能要素のサイズが十分か（最小44x44px）。要素同士の間隔が狭すぎず、誤タップを誘発しないか。
   - **A2HS（ホーム画面追加）案内**: A2HSを促す案内バナーやダイアログの表示タイミング、iOS/Androidそれぞれのデザインが仕様書通り（ユーザーに不快感を与えない自然な誘導）になっているか。
   - **オフライン時のフィードバック**: 通信切断（オフライン）状態においても、ローカルキャッシュ（OPFS含む）からロードされた画面やカードが崩れず、適切なオフラインインジケーターや代替UIが表示されているか。

## ワークフロー

### Step 1: E2Eスクリーンショットの目視監査（静的QA）
E2Eテストの実行結果として保存されている画像ファイルを確認します。
1. `tests/screenshots/` ディレクトリ内の `.png` 画像をリストアップします。
2. `view_file` ツールを使用して、画像を直接開きます。
3. 特にライトモードとダークモードの比較画像、およびモバイルデバイス解像度でキャプチャされた画像（もしあれば）を並べてチェックし、「テーマの整合性」と「視認性」のバグを探します。

### Step 2: 実機SandboxでのLLM挙動監査（動的QA）
ローカルのブラウザSandbox環境を立ち上げ、実際のLLMとやり取りを行い、動的なUXを監査します。
1. `sandbox-mcp-debugging` または `chrome-devtools-mcp` スキル/ツールを使用して、ヘッドレスChromeセッションに接続します。
2. アプリケーションのLLM（WebLLM または LiteRT-LM）の初期化プロセスとプロンプト入力を実行します。
3. 「ダウンロード中のインジケーター」や「ストリーミング中のテキストの挙動」を視覚的に（DOMの状態や表示結果を通して）観察し、崩れや不自然な動きがないか確認します。
4. モバイルPWAのE2Eテスト環境（Playwright）でService Workerの登録スコープやキャッシュの有効性、オフライン時の画面表示崩れがないかを確認します。

### Step 3: 視覚的バグとロギング不足のIssue起票
発見したレイアウト崩れやテーマ不整合、または「ログ仕込み漏れ」について、GitHub Issueを起票します。

**ログ不足の指摘例**:
- 「Playwright上で特定のエラーUIが表示されたが、コンソールやトレースに原因を追跡するための十分なエラーログが出力されていません。`logger.error` などのログ出力を追加してください。」

```bash
gh issue create --title "bug: [Dark Mode] 設定画面のボタンが白背景のままになる" --body "### 視覚的バグの詳細
ダークモード設定時においても、設定画面下部の保存ボタンが白背景のままであり、テキスト（白文字）が読めなくなっています。

### 再現箇所
tests/screenshots/settings-dark.png の右下部分。

### 期待されるUX
ダークモード用の \`dark:bg-gray-800\` のようなクラスが欠落していると思われます。正しく反転させてください。" --label "bug,ux,status: triage"
```
**ルール**:
- 発見した視覚的バグごとに、別々のIssueを起票することを推奨します。
- 可能であれば、`gh-image` CLI拡張を使って、崩れている部分のスクショ画像をIssueに添付してください。

### Step 4: 監査レポートの提出
- 全ての監査が完了したら、`post-session-report` スキルを呼び出します。
- ロールに `QA Visual Auditor` を指定し、チェックした画像の枚数、SandboxでのLLMの挙動調査結果、そして新たに起票した視覚的バグのリストをMarkdownで提出します。
